Deconflict

Browser-based WiFi planner with signal heatmaps, multi-floor support, and graph coloring channel assignment

Live Demo Source

WiFi works until it doesn't. You add a second access point and suddenly both are slower. The problem is almost always channel conflict: two APs close enough to interfere, broadcasting on the same or overlapping frequencies. Deconflict is a browser-based tool for planning around this. Upload a floorplan, drop access points where you want them, and the app assigns channels automatically using graph coloring. It models multi-floor buildings with per-material wall and slab attenuation, and includes a database of real AP models so the signal estimates reflect actual hardware.

Channel assignment as graph coloring

The channel assignment problem maps directly to graph coloring. Each AP is a node. If two APs are close enough to interfere (their coverage radii overlap), they share an edge. Assigning channels so no adjacent nodes share a color is the classic vertex coloring problem. The solver uses DSatur (Degree of Saturation), which picks the most constrained node first, assigns it the lowest available channel, and repeats. For WiFi planning this works well because the densest clusters get colored first, and DSatur typically finds near-optimal solutions in under 5 milliseconds.

Each band is solved independently, since 2.4 GHz, 5 GHz, and 6 GHz don't interfere with each other. The solver builds a separate interference graph per band, each with its own channel pool, then merges the results. The app also ships with three alternative algorithms for comparison: Welsh-Powell (sorts by degree), greedy (simplest possible), and backtracking (finds the true minimum chromatic number but is slower). All run in a Web Worker so the canvas stays responsive, and the solver re-runs automatically every time you add, move, or remove an AP.

Multi-floor support

Real buildings have more than one floor, and an AP on the second floor absolutely interferes with one directly below it on the first. Deconflict models this. Each floor has its own floorplan, wall mask, and set of APs, but the solver and heatmap renderer see all floors at once. When you're viewing floor 2, APs from floors 1 and 3 are projected as virtual APs with attenuated signal. The 3D distance calculation accounts for ceiling heights and slab thickness between floors, and the signal passes through each intervening floor slab with material-specific attenuation.

Four floor slab materials model different construction types: wood frame (8 dB per slab at 5 GHz), concrete (20 dB), reinforced concrete (28 dB), and steel deck (35 dB). These values scale with thickness and vary by band, using per-meter dB rates derived from ITU-R P.1238. Each floor can have a different slab material, so you can model a building with a concrete ground floor and wood-framed upper stories. The floor switcher is a horizontal pill strip in the sidebar: click a pill to switch floors, double-click to rename, and click + to add a new one. Each floor can have its own ceiling height, slab thickness, and slab material.

AP vendor database

The app includes a database of 101 real access point models from 14 vendors: Ubiquiti, TP-Link, Netgear, Ruckus, Cisco Meraki, Aruba, ASUS, Linksys, Cisco, eero, Google, Apple, Fortinet, and EnGenius. When you select a model, its per-band specifications auto-fill: TX power from manufacturer datasheets and FCC filings (typically 17-25 dBm), supported bands, maximum channel widths (20 MHz up to 320 MHz on WiFi 7 models), MIMO stream count, and typical indoor range. The model picker is a searchable combobox that fuzzy-matches across vendor, model name, and WiFi standard.

Throughput estimation uses model-specific MIMO streams and PHY rate tables spanning WiFi 4 through WiFi 7. A 2x2 MIMO AP on 80 MHz WiFi 6 gets a different base rate than a 4x4 on 160 MHz WiFi 7. Co-channel contention is calculated from real attenuated signal strength between AP pairs (accounting for walls and floor slabs), not geometric overlap. If two APs share a channel, the effective throughput for each drops proportionally to how much they actually interfere.

Wall materials

Not all walls attenuate equally. A glass partition barely dents the signal. A concrete wall kills it. The app models six wall materials with thickness-aware attenuation based on ITU-R P.1238: drywall (3 dB), glass (2 dB), wood (5 dB), brick (8 dB), concrete (12 dB), and metal (20 dB). Each material stores a dB-per-meter rate and a typical thickness, so the ray marcher accumulates loss per pixel of wall traversed rather than applying a flat penalty per crossing. After the auto-detector finds walls, you can click any wall segment to override its material, or use the material paint brush to assign materials across regions.

The RF model uses DDA ray marching to trace rays from each AP through the wall grid, accumulating per-material dB losses at each step. A precomputed attenuation field stores O(1) lookup values on a 6-pixel grid (12-pixel during drag for responsiveness). The heatmap updates live as you draw or erase walls, re-syncing wall data on each brush stroke.

Signal heatmap

The heatmap renders to an offscreen canvas using inverse quartic path loss: signal strength falls off as 1/(1+(d/r)^4), where d is distance and r is the AP's range. For each cell, the renderer evaluates all APs on the current floor plus virtual projections from other floors, calculates signal strength with 3D distance for cross-floor APs, and traces through walls to accumulate material-specific attenuation. Floor slab loss is applied separately using an ITU aggregate model. The result maps to five color bands from green (excellent) to dark red (dead zone).

Wall detection runs automatically when you load a floorplan. The pipeline starts with OCR via Tesseract.js to mask out text labels (room names, dimensions) that would confuse the detector. It then applies adaptive local thresholding using an integral image, morphological close to seal door gaps, flood fill to identify the building interior, and blob filtering to remove noise. Walls that the algorithm misses can be drawn manually with the wall brush.

Wall cleanup brush

The auto-detector gets you most of the way, but some walls need manual correction. The wall editor has three modes: draw (adds walls with the active material), erase (removes walls), and material paint (changes the material of existing walls without drawing new ones). The brush is circular with configurable radius, and strokes are interpolated so you get smooth lines even with fast mouse movement. On mobile, single-finger touch draws while two-finger gestures pan and zoom. The heatmap updates live as you paint.

Wall masks use a sparse tiled storage system (256x256 pixel tiles, allocated on first write) so the canvas can extend infinitely without pre-allocating a massive pixel buffer. Tiles are flattened to a Uint8Array for the hot-path consumers (ray marcher, renderer, flood fill) and cached when bounds haven't changed.

Automatic placement optimizer

You can place APs manually, but the app also includes a three-stage optimizer that adjusts positions for better coverage, all running in a Web Worker.

Stage one is signal-weighted Lloyd's algorithm (k-means). It seeds positions using k-means++ initialization, then runs 30 iterations where each sample point is assigned to its strongest AP, with weak-signal areas weighted 3x higher so the optimizer pulls APs toward dead zones. Stage two is particle swarm optimization with 20 particles over 60 iterations, exploring the neighborhood around the Lloyd's solution with decreasing inertia (0.9 to 0.4). Stage three is coordinate descent polish, searching 8 directions at progressively finer steps (5 pixels down to 0.5) until no improvement is found.

All three stages share a precomputed wall attenuation cache: a Float32Array storing the dB loss from every source grid cell to every sample point, built once before optimization starts. When an AP moves, only its column gets recomputed. In multi-floor buildings, APs on other floors are treated as fixed signal sources with cross-floor attenuation, so the optimizer accounts for their contribution without trying to move them. A real-time coverage bar in the sidebar shows the current percentage as you work.

The RF model

The app covers all three consumer WiFi bands: 2.4 GHz (channels 1-13), 5 GHz (UNII-1 through UNII-3, with DFS channel flags), and 6 GHz (channels 1-233, with preferred scanning channel subsets). Channel widths range from 20 MHz up to 320 MHz on 6 GHz. Reference path loss varies by band: 40 dB at 2.4 GHz, 47 dB at 5 GHz, 49 dB at 6 GHz. The receiver sensitivity threshold is -85 dBm, a typical WiFi client.

The model is simplified on purpose. Real RF propagation depends on furniture, reflection paths, and antenna patterns. Deconflict uses per-material wall attenuation, thickness-aware floor slab loss, and free-space path loss because the goal is channel planning, not site survey accuracy. The heatmap tells you "this corner has weak coverage" and "these two APs are conflicting," which is what you need to decide where to put the next AP.

Implementation

The frontend is Svelte 5 with SvelteKit, rendering to HTML5 Canvas with a custom layered engine (floorplan, walls, heatmap, APs, selection UI). State is managed with Svelte 5 runes split into reactive atoms, with per-floor state (floorplan, walls, APs) stored independently and synced on floor switch. The canvas uses matrix transforms for pan and zoom, redraws on a dirty flag to avoid unnecessary frames, and scales for retina displays. State auto-saves to localStorage with undo/redo support, and the save format migrates automatically from older single-floor schemas. You can also drag-and-drop a floorplan image directly onto the canvas.

The codebase is a monorepo managed by pnpm and Turborepo. The solver, geometry, and channel packages are separate TypeScript libraries with their own test suites. The app builds to static files via SvelteKit's static adapter and deploys to Cloudflare Pages, with PR branches auto-deploying to preview URLs. A service worker enables offline use and PWA installation.
